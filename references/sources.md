# 数据源策略

## 重要背景（用户提供的领域知识）

中国大陆校园招聘信息的发布路径通常是：**公司先通过自己的招聘官方微信公众号（一般名为"XX招聘"/"XX校园招聘"）发布"秋招正式启动"的推文**，推文里会附上跳转到**公司官方招聘官网/招聘系统**（如 zhaopin.xxx.com、campus.xxx.com、或第三方 SaaS 如北森/大易/Moka 搭建的官网）的链接，官网上才有完整的岗位列表和详细要求。第三方聚合平台（牛客网、猎聘校园等）上的信息大多是转载或用户自发讨论，时效性和完整度不如"公众号公告 → 官网详情"这条一手链路。

因此发现层应该把"搜索各公司招聘公众号的秋招公告"当作和聚合平台同等重要、甚至更优先的信息源，而不只是核实层的兜底手段；核实层"访问官方招聘官网"这一步也不只是给聚合平台的信息找权威链接，而是所有渠道发现的岗位都应该尽量拿到官网这个最终详情页链接。

## 发现层（并行，每次运行 4 个子 agent）

按优先级排列：

1. **牛客网**（nowcoder.com）求职/校招板块 — 应届生一手讨论区，信息量最大。
2. **公司招聘微信公众号 + 官网详情**（经 WebSearch/CDP 搜索"XX招聘"公众号发布的"2026秋招/秋季校园招聘正式启动"类推文，再跳转到文中的官方招聘官网核实详细岗位）— 一手信息源，优先级高。

   > **2026-07-08 更新**：应届生求职网（yjbys.com）经实测已停用校园招聘频道（只剩简历模板/求职百科内容，历史招聘专题页返回 404/403），**不再作为数据源**，此位置替换为"公司招聘公众号+官网"。
3. **51job校园招聘**（campus.51job.com）+ **智联招聘校园**（xiaoyuan.zhaopin.com）+ **猎聘校园**（campus.liepin.com）— 结构相似，合并为一个子任务，用 `config.json` 里 `job_category_label` 的值作为频道内筛选词。
4. **门户/公众号"秋招名单/时间表"汇总贴**（经 WebSearch 发现，如搜索"{target_season_label} 名企名单"、"校招进行时 {job_category_label}"，把 config.json 里的实际值代入）+ **实习僧**（shixiseng.com）— 最快知道"新增开秋招的公司"，但属二手转载，需核实层核实。

## 核实层（顺序执行，仅针对本次新出现的公司）

对 state 中全新出现的公司（即 `state/seen_postings.json` 里没有记录过的公司名），访问其**官方校园招聘官网**核实岗位、获取权威链接替换聚合帖/公众号链接。每次运行最多核实 5 家新公司，避免每日成本无限增长。已在 state 中出现过的公司不重复核实。

## 明确降级 / 不作为每日默认来源

**BOSS直聘、拉勾** — 需要登录态 + CDP 才能稳定访问，反爬和账号风险较高。只有当发现层 4 个来源当天完全没有结果时，才考虑作为补充手段临时启用，不写入常规每日流程。

## 兜底原则

某个来源当天打不开、或没有相关结果，对应子 agent 应如实汇报为空（例如"该来源今日无可用目标方向校招信息"），**不要**反复重试同一种访问方式去"确认"是否真的没有。orchestrator 不应因为单一来源当天失败/为空就中断整体流程，其余来源的结果照常合并处理。

## 调用约定

每个发现层子 agent 的 prompt 必须：
- 显式包含「必须加载 web-access skill 并遵循指引」这句话。
- 用目标性措辞下达任务，把 `config.json` 里 `job_category_label`、`target_grad_year` 的实际值代入（例如"调研牛客网上有哪些{job_category_label}方向的{target_grad_year}届秋招正式岗位"），不要指定具体方法动词（不要写"用WebSearch搜索"之类），把搜索/抓取/CDP升级的路由决策留给 web-access 自己判断。
- 要求返回结构化列表，字段：公司名称、职位名称、工作城市、岗位亮点摘要（1句话）、原始链接（完整含参数）、来源栏目名称。
